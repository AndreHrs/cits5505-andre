Close All HTML Elements

In HTML, you do not have to close all elements (for example the element). It is completely possible to do this 


                    <section>
                      <p>This is a paragraph.
                      <p>This is a paragraph.
                    </section>

But it is best practice to close all HTML elements. [1] This will make debugging easier and help prevent issues with your code. It will also be beneficial if you are using linter or code formatter tools.

Avoid Long Code Lines

When using an HTML editor, it is NOT convenient to scroll right and left to read the HTML code. Try to avoid too long code lines and add line break when it nearly reaches the right side limit of your screen. Google's HTML Formatting Style Guide recommends breaking lines at 80 characters unless it's a URL or the metadata tags that can be as long as they need.[3]

Decouple HTML/Javascript

Keep HTML and JavaScript in separate files to maintain a clean, modular codebase. HTML should focus on defining structure and content, while JavaScript manages behavior. Avoid inline scripts and large chunks of HTML generated by JavaScript, as this mixing makes debugging and maintenance more challenging. By using external JavaScript files and attaching behaviors through the DOM, you improve readability, simplify error tracking, and make your code more scalable and robust.[4, pp 960-961]

Prefer CSS over JavaScript

If something we want to do with JavaScript can be done with CSS instead, we use CSS. Before we try to add JavaScript and even when we add JavaScript, we should consider having CSS making the most of the style and using JavaScript for things like triggers and side effects. In general, our applications are faster using CSS instead of JavaScript.[5]

Code below is considered bad practice because you are using javascript unnecessarily for things that can be done with css 


                  <button id="badButton">Hover over me</button>
                  <script>
                    document.getElementById("badButton").addEventListener("mouseenter", function () {
                      this.style.backgroundColor = "blue";
                    });

                    document.getElementById("badButton").addEventListener("mouseleave", function () {
                      this.style.backgroundColor = "gray";
                    });
                  </script>

Example of better practice to use css


                  <button class="good-button">Hover over me</button>

                  <style>
                    .good-button {
                      background-color: gray;
                    }

                    .good-button:hover {
                      background-color: blue;
                    }
                  </style>

Avoid qualified selectors

Basically, selectors who are needlessly prepended by an element. Removing unnecessary elements makes your CSS more reusable, simpler, and faster because it reduces specificity and browser workload.[5] 


                  /* Bad practice, only specific for ul 
                  with class nav and div with class header */
                  ul.nav {}
                  div.header {}
                  
                  /* Better practice, can be used by 
                  other elements other than ul and div */
                  .nav {}
                  .header {}

Avoid dangerous selectors

A dangerous selector is one with far too broad a reach. An obvious and simple example of a dangerous selector might be: 


                  /*dangerous, affects all div elements*/
                  div {
                    background-color: #ffc;
                  }

To give such specific styling to such a generic selector is dangerous. Our styles will leak out into areas they shouldn't as soon as we start trying to use that element again. We'll need to start undoing styles (adding more code to take styles away) in order to combat this. [5]

Use Hex Code instead of Name Color

It has been pointed out by experts and professionals that when using "hex code", they found that it is faster for 4-5 runs. So rather than using the "name color", go for "#hex code" instead. Besides that, if you work in a team, UI Designer often will have their colors in hex code too.[5]

Use CSS variables

Complex websites have very large amounts of CSS, and this often results in a lot of repeated CSS values. For example, it's common to see the same color used in hundreds of different places in stylesheets. Changing a color that's been duplicated in many places requires a search and replace across all rules and CSS files. Custom properties allow a value to be defined in one place, then referenced in multiple other places so that it's easier to work with. Another benefit is readability and semantics. For example, --main-text-color is easier to understand than the hexadecimal color #00ff00, especially if the color is used in different contexts.[6]

Avoid using var and be scope aware

As the scope chain grows longer, accessing variables outside the current scope takes more time. Global variables are always slower to access than local ones because the entire scope chain must be traversed. Additionally, using the var keyword can unexpectedly create variables due to JavaScript's function-level scoping, which can introduce bugs and make code harder to understand. In modern JavaScript, there's rarely any reason to choose var over const or let—both of which offer block-level scoping. [4, pp 967-968]

JavaScript can run code and generate HTML directly from strings. For example

Initialize variable

JavaScript's loosely typed nature means it's easy to lose track of what kind of data a variable is meant to hold, even with good naming practices. One practical way to signal a variable's intended type is through its initialization. For example, if a variable should hold a Boolean value, you initialize it with either true or false; similarly, a variable meant for numbers should be given a numeric value. Consider these examples 


                  let found = false; // Boolean
                  let count = -1;    // number
                  let name = "";     // string
                  let person = null; // object

Initializing variables like this gives an immediate hint about the type of data they're expected to contain, even though this method can't be used with function arguments.[4, p. 958]

Trivia

Javascript and Python's loosely typed nature is often called duck typing. Duck typing came fromt he term if it quacks like a duck, it can be treated as a duck. Meaning that if an object behaves like what you need, then you can use it—regardless of its actual type .

Decouple Application Logic and Event Handlers

Separate your core logic from event handlers so each part focuses on its own task. An event handler should capture only what's needed from the event object and pass that data to a dedicated function that handles the logic. This approach makes debugging simpler, allows you to reuse the same logic for multiple events, and makes your code more testable and maintainable. If you ever need to trigger the logic without an event (for testing or automation), you can do so easily by calling the logic function directly. 


                  // Bad practice, mixing logic and event handling
                  // Hard to test and reuse
                  function handleKeyPress(event) {
                    if (event.keyCode == 13) {
                      let target = event.target;
                      let value = 5 * parseInt(target.value);
                      if (value> 10) {
                      document.getElementById("error-msg").style.display = "block";
                      }
                    }
                  }

                  // Better practice, separating logic from event handling
                  // Easy to test and reuse
                  function validateValue(value) {
                    value = 5 * parseInt(value);
                    if (value> 10) {
                      document.getElementById("error-msg").style.display = "block";
                    }
                  }
                  function handleKeyPress(event) {
                    if (event.keyCode == 13) {
                      let target = event.target;
                      validateValue(target.value);
                    }
                  }

Code Readability and comments

For code to be maintainable, it must be readable—this involves not only proper indentation but also effective use of comments. In Professional JavaScript for Web Developers , Matt Frisbie advises that every function or method should include a comment that explains its purpose and, if necessary, the algorithm it employs.[4, p. 957] However, as Robert C. Martin points out in Clean Code, overreliance on comments often indicates that the code itself is unclear. He argues that instead of adding comments to explain messy code, you should clean it up. Essentially, if your code is clear and self-explanatory, you won't need extra comments [7, p. 55].

What is Linter?

Linter or Linting tools are automated tools that analyze source code to detect potential errors, security vulnerabilities, or coding style issues. [9]It will greatly helps your code quality. Especially when we are using javascript where weird things can happen.

The original lint tool was designed to analyze C source code for potential errors and stylistic issues. Since then, linting tools have evolved to work with a variety of programming languages, including JavaScript, Python, and Ruby.

Linting tools are useful for a number of reasons. Firstly, they help you catch errors early in the development process, when they are easier and cheaper to fix.

Secondly, they can help promote coding standards and best practices within a development team, ensuring that code is consistent and maintainable.

Finally, they can help you identify potential security vulnerabilities in your code, reducing the risk of a breach.[9]

One of the popular linting tools for Javascript is ESLint. Which you can also get as VS Code extension here .

What is Code Formatter?

Code formatter are tools that help you format source code automatically. It is generally accepted that having a common style guide is valuable for a project and team.[10]

I want to write code. Not spend cycles on formatting.[10] Is one of the motivation to adopt a code formatter.

Most of the time readability has something it do with indentation of code and code style:

  1. javascript accepts both single quotes and double quotes. Which one should you use across the whole project
  2. how many indentations are needed? 2 spaces? 4 spaces?
  3. sometimes I forgot to add semicolon
  4. after adding some arguments to function call, it exceeds 80 characters and it's a pain to line break it manually

Adopting code formatter such as Prettier will enforces a consistent code style across all your code base.[11] Group project with multiple collaborators will greatly benefit from the consistent styling. This can be easily done by sharing the prettier configuration such as in the git repository.
